**此文档仅做设想，请勿在构建或修改代码时应用本文档的设计**

## 一、目标

在现有 OIDC 单点登录基础上，增加本地用户管理体系，支持以下场景：

1. **纯本地模式**：不依赖外部 OIDC 提供商，使用邮箱+密码登录
2. **纯 OIDC 模式**：保持当前架构不变
3. **混合模式**：OIDC 登录与本地登录并存；OIDC 用户可自行设置本地密码，断开 OIDC 后仍可登录
4. **自注册**：允许用户在登录页自行注册账户（可选开关，管理员控制）

---

## 二、认证模式选择（Setup 阶段）

在首次配置页面中，OIDC 提供商选择扩展为 4 个选项：

| 选项 | 说明 | 初始管理员来源 |
|------|------|---------------|
| Keycloak | 与当前一致 | 首个 OIDC 登录用户 |
| Auth0 | 与当前一致 | 首个 OIDC 登录用户 |
| 通用 OIDC | 与当前一致 | 首个 OIDC 登录用户 |
| **本地用户管理** | 系统自行管理用户认证 | Setup 表单中创建的管理员账户 |

- 选定后 `provider_type` 在 Setup 完成后**只读**，不支持运行时切换（避免已有用户的认证方式衔接复杂度爆炸）。
- 但**混合模式**通过另一种方式实现：选择任意 OIDC 提供商后，可在管理后台开启"允许本地登录"和"允许自注册"开关。

## 三、混合模式（OIDC + 本地登录并行）

### 3.1 启用方式

管理员在后台系统配置页面开启两个独立开关：

- **允许本地登录**（`local_auth_enabled`）：开启后，所有已设有本地密码的用户可使用邮箱+密码登录
- **允许自注册**（`self_registration_enabled`）：开启后，登录页显示注册入口，用户可自行创建账户。依赖"允许本地登录"已开启

### 3.2 用户身份模型

不再引入 `auth_provider` 列。登录能力由**是否设有密码**决定：

| 场景 | user_id | email | password_hash | 登录方式 |
|------|---------|-------|---------------|---------|
| OIDC 用户（未设密码） | OIDC sub | OIDC email | NULL | 仅 OIDC |
| OIDC 用户（已设密码） | OIDC sub | OIDC email | bcrypt hash | OIDC 或邮箱+密码 |
| 本地用户（管理员创建） | UUID v4 | 管理员填写 | bcrypt hash | 邮箱+密码 |
| 自注册用户 | UUID v4 | 注册邮箱 | bcrypt hash | 邮箱+密码 |

### 3.3 email 唯一性约束

- `users.email` 增加 UNIQUE 约束
- OIDC 登录时，若 `sub` 对应的用户不存在但 email 已被占用 → 视为同一用户，绑定 OIDC 身份（更新 `user_id` 为 OIDC sub）
- 管理员创建本地用户时，若 email 已被占用 → 返回 409 提示
- 自注册时，若 email 已被占用 → 提示"该邮箱已被注册"

### 3.4 OIDC 断开后的用户存续

管理员断开 OIDC（如提供商不可用或主动移除配置）后：

- OIDC 的 `/auth/login` 重定向失效
- 已设了本地密码的 OIDC 用户不受影响 → 可用邮箱+密码登录
- 未设密码的 OIDC 用户暂时无法登录 → 管理员可通过 UserManage 为其设置密码（系统自动发送重置邮件）
- 提供批量操作：「为所有无密码用户生成重置链接并发送邮件」

## 四、管理员初始化的改进

**当前逻辑**（AGENTS.md 5.x）：`admin_initialized=true` 后，即使 `users` 表被清空也不再产生新管理员。

**改进逻辑**：

- `admin_initialized` 标记仍然存在，用于防止已有用户时意外产生第二个管理员
- 但当 `users` 表为空（即 0 行用户记录）时，忽略 `admin_initialized` 标记，下一个创建的用户自动成为管理员
- 这意味着：如果数据库被重置清空 / 迁移到新环境 / 所有用户被删除，系统可以重新开始

此改进适用于所有模式（OIDC / 本地 / 混合）。

## 五、自注册流程

### 5.1 开启条件

- 管理员在后台开启"允许自注册"
- 依赖"允许本地登录"已开启

### 5.2 注册流程

```
登录页 → 点击「注册」→ 注册页
  → 填写 用户名 + 邮箱 + 密码（+ 确认密码）
  → 提交 → 后端校验
    → email 未被占用 → 创建用户 (role=user, user_id=UUID v4)
      → 若 users 表为空 → 该用户 role=admin
    → 发送欢迎邮件（可选）
  → 自动登录 → 跳转首页
```

### 5.3 管理员对自注册用户的控制

- UserManage 列表中标注用户来源（OIDC / 本地 / 自注册）
- 管理员可禁用/删除自注册用户，与普通用户管理一致
- 关闭"允许自注册"后，不影响已注册用户，仅隐藏注册入口

## 六、用户个人设置页面（新增）

路由：`/profile`，所有登录用户可访问。

### 6.1 功能清单

| 功能 | 说明 |
|------|------|
| 查看个人信息 | 用户名、邮箱、角色、订阅级别 |
| 修改邮箱 | 需验证新邮箱未被占用；修改后需重新登录 |
| 设置/修改本地密码 | OIDC 用户可在此绑定本地密码；本地用户可修改密码 |
| 记住我设备管理 | 查看/撤销"记住我"会话（远期） |

### 6.2 设置密码的安全性

- 设置密码前需先验证当前密码（若已设有密码）
- OIDC 用户首次设置密码无需验证旧密码（没有旧密码），但需通过 OIDC 会话确认身份（即仅在已登录状态下可设）

## 七、邮件服务

### 7.1 用途

| 场景 | 邮件类型 |
|------|---------|
| 忘记密码 | 密码重置链接（含一次性 token，1 小时有效） |
| 管理员重置用户密码 | 同上，管理员触发后系统发送 |
| 自注册欢迎 | 欢迎邮件（可选，默认关闭） |
| 邮箱变更确认 | 验证新邮箱所有权（远期） |

### 7.2 SMTP 配置

管理员在后台系统配置页面配置 SMTP 参数。若未配置 SMTP，密码重置功能降级为：
- 管理员在 UserManage 直接设置用户密码（不发送邮件，口头告知）
- 忘记密码页面显示"请联系管理员重置密码"

### 7.3 密码重置流程

```
登录页 → 点击「忘记密码」
  → 输入邮箱
  → 系统生成重置 token（password_reset_tokens 表），发送邮件
  → 无论邮箱是否存在，页面均提示"若该邮箱已注册，重置链接已发送"
  → 用户点击邮件中的链接 → 重置页面（验证 token 有效性）
  → 输入新密码 → 完成 → 自动登录
```

## 十、API 端点规划（概要）

仅列端点用途，不列具体实现。

| 端点 | 用途 | 认证 |
|------|------|------|
| `POST /auth/local/login` | 本地登录 | 无（速率限制） |
| `POST /auth/local/register` | 自注册 | 无（速率限制，需开关开启） |
| `POST /auth/forgot-password` | 发送重置邮件 | 无（速率限制） |
| `POST /auth/reset-password` | 执行密码重置 | 无（token 验证） |
| `POST /user/profile/password` | 当前用户设置/修改密码 | JWT |
| `PUT /user/profile` | 当前用户修改个人信息 | JWT |
| `GET /user/profile` | 获取个人信息 | JWT |
| `POST /admin/users` | 管理员创建本地用户 | JWT + Admin |
| `POST /admin/users/:id/reset-password` | 管理员为用户重置密码 | JWT + Admin |
| `PUT /admin/system/local-auth` | 配置本地登录/自注册开关 | JWT + Admin |
| `PUT /admin/system/smtp` | 配置 SMTP | JWT + Admin |

## 十一、数据库变更概要

## 十二、未决事项

以下问题在进入实施阶段前需要最终决策：

1. **OIDC 断开的触发条件**：是管理员手动"断开 OIDC"（清空配置），还是系统检测到提供商不可用自动降级？建议手动触发，避免误判。

2. **自注册的审批机制**：是否需要管理员审批后才能登录？还是注册后立即可用？建议初期无需审批，注册即用。

3. **「记住我」的撤销**：是否需要用户能查看和撤销"记住我"的设备？远期可在 Profile 页实现，初期可跳过。

4. **本地模式 Setup 的管理员密码复杂度要求**：当前 OIDC 模式不涉及。建议 ≥8 字符。

5. **密码重置 token 的有效期**：建议 1 小时。与 OIDC state 的 10 分钟 TTL 相比更长，因为邮件到达有延迟。
